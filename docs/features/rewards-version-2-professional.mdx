# SEND Rewards Program

The SEND rewards program distributes tokens monthly to qualified participants who actively circulate SEND within the ecosystem. This document explains the qualification criteria, reward calculation methodology, and key concepts.

---

## Qualification Requirements

To qualify for monthly rewards, participants must meet all three criteria at the snapshot timestamp:

| Requirement | Description |
|-------------|-------------|
| Sendtag Registration | A verified identity tag linked to your account |
| Minimum SEND Balance | Hold the required minimum SEND tokens |
| Send Earn Deposit | Maintain the minimum USDC balance in Send Earn |

```
                    QUALIFICATION FLOW

    ┌─────────────┐     ┌─────────────┐     ┌─────────────┐
    │   Sendtag   │────▶│   Min SEND  │────▶│  Send Earn  │
    │   Required  │     │   Balance   │     │   Deposit   │
    └─────────────┘     └─────────────┘     └─────────────┘
            │                  │                   │
            └──────────────────┴───────────────────┘
                              │
                              ▼
                    ┌─────────────────┐
                    │    QUALIFIED    │
                    │   for rewards   │
                    └─────────────────┘
```

Missing any requirement disqualifies the user for that distribution period.

## Reward Pool Structure

The monthly distribution amount is allocated across two pools:

```
    ┌──────────────────────────────────────────────────────┐
    │              TOTAL DISTRIBUTION AMOUNT                │
    └──────────────────────────────────────────────────────┘
                              │
              ┌───────────────┴───────────────┐
              ▼                               ▼
    ┌─────────────────────┐         ┌─────────────────────┐
    │     FIXED POOL      │         │    HODLER POOL      │
    │                     │         │                     │
    │  Calculated first   │         │  Remaining amount   │
    │  from task values   │         │  after fixed pool   │
    │  and multipliers    │         │                     │
    │                     │         │  Distributed by     │
    │  Capped by:         │         │  weighted balance   │
    │  hodler + score +   │         │  (sigmoid curve)    │
    │  ticket purchases   │         │                     │
    └─────────────────────┘         └─────────────────────┘
```

**Note:** The bonus pool referenced in earlier versions is always set to zero in the current implementation.

## Send Score and Send Ceiling

The send score mechanism incentivizes active token circulation while preventing abuse.

### Send Ceiling Calculation

```
                        SEND CEILING FORMULA

    Previous Distribution Share (or hodler_min_balance)
    ─────────────────────────────────────────────────────
            minimum_sends × scaling_divisor

                            =

                    Per-Recipient Cap
```

### Send Score Accumulation

```
    ┌──────────────────────────────────────────────────────────┐
    │                  SEND SCORE PROCESS                       │
    │                                                           │
    │   For each recipient:                                     │
    │                                                           │
    │   1. Check recipient has earn_min_balance  ─▶ Required    │
    │                                                           │
    │   2. Sum all transfers to that recipient                  │
    │                                                           │
    │   3. Cap at send_ceiling                                  │
    │                                                           │
    │   Total Score = Σ min(transfers_to_recipient, ceiling)    │
    │                                                           │
    └──────────────────────────────────────────────────────────┘
```

Transfers to recipients without adequate Send Earn balances do not count toward the send score.

## Slash Percentage and Reward Scaling

Rewards scale based on sending activity. Users who do not send receive reduced or zero rewards.

```
                    SLASH CALCULATION

    ┌────────────────────────────────────────────────────────┐
    │                                                         │
    │   scaled_previous_reward = previous_reward              │
    │                            ÷ scaling_divisor            │
    │                                                         │
    │   capped_weight = min(send_ceiling_weight,              │
    │                       scaled_previous_reward)           │
    │                                                         │
    │   slash_percentage = capped_weight                      │
    │                      ÷ scaled_previous_reward           │
    │                                                         │
    │   If send_ceiling_weight = 0  ──▶  slash = 0%           │
    │   Both fixed and hodler rewards become zero             │
    │                                                         │
    └────────────────────────────────────────────────────────┘
```

## Task Verification System

Tasks contribute fixed token amounts to your reward. Each task type has defined parameters:

| Component | Description |
|-----------|-------------|
| fixed_value | Base token amount per task completion |
| weight | Number of completions or task-specific value |
| multiplier | Scaling factor for accumulated completions |

### Multiplier Mechanics

```
    ┌────────────────────────────────────────────────────────┐
    │                  MULTIPLIER SYSTEM                      │
    │                                                         │
    │   Starting value:     multiplier_min (often 1.0)        │
    │   Increment:          multiplier_step per completion    │
    │   Maximum:            multiplier_max                    │
    │                                                         │
    │   Example with step=0.1, max=2.0:                       │
    │                                                         │
    │   Completion 1:  1.0×                                   │
    │   Completion 2:  1.1×                                   │
    │   Completion 3:  1.2×                                   │
    │   ...                                                   │
    │   Completion 11+: 2.0× (capped)                         │
    │                                                         │
    │   Final = base_fixed_amount × Π(all_multipliers)        │
    │                                                         │
    └────────────────────────────────────────────────────────┘
```

**Important:** Multipliers and task completions are per-distribution. They do not carry forward to subsequent months.

## Fixed Reward Cap

To maintain economic balance, fixed pool rewards are capped:

```
    Fixed Reward ≤ Initial Hodler Amount + Send Score + Ticket Purchase Value
```

Where:
- **Initial Hodler Amount**: Your share from the time-adjusted hodler calculation
- **Send Score**: Total capped sends to eligible recipients
- **Ticket Purchase Value**: SEND spent on Sendpot tickets

---

## Progress Indicator

The application displays a progress percentage calculated as:

```
    Progress = min(send_ceiling_weight, scaled_previous_reward)
               ───────────────────────────────────────────────
                         scaled_previous_reward
```

This represents the proportion of earning potential unlocked through sending activity, not the raw amount of SEND transferred.

---

## Summary

The SEND rewards program allocates monthly distributions based on:

1. **Qualification**: Sendtag, minimum SEND, and Send Earn deposit
2. **Sending Activity**: Send score determines reward scaling
3. **Task Completion**: Fixed amounts with optional multipliers
4. **Balance Weighting**: Hodler pool distributed by sigmoid-weighted balances

All metrics reset each distribution period. Consistent monthly participation is required to maintain reward eligibility.
